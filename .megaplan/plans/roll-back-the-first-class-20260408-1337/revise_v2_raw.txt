        Idea:
Roll back the "first-class shot groups with projection layer" architecture introduced in a previous megaplan. Replace it with a simpler "shot group is a soft tag, cohesion is enforced by edit operations" model. The current projection-based architecture has produced a recurring class of bugs that we cannot fix incrementally — every patch surfaces another bug because the underlying model is wrong. This refactor eliminates the entire bug class structurally.

## Why (the structural problem)

The previous "first-class shot groups" megaplan introduced a pinnedShotGroups[i] = { shotId, trackId, start, children: [{clipId, offset, duration}], clipIds, mode, ... } data shape. Cohesion is enforced by re-deriving clip positions from pinnedShotGroups[].children via projectPinnedShotGroups, which runs three times per commit cycle (inside resizeGroupEdge, inside applyEdit, and inside buildDataFromCurrentRegistry via repairConfig).

This is "cohesion by re-projection," which means:

1. Two parallel data shapes for the same thing. pinnedShotGroups[i].children[j].duration vs clips[k].hold. Any edit that bypasses the projection (a free-clip resize fallback, a stale closure, a missing prop) gets silently reverted because the next projection re-derives from children.

2. Auto-save fires during drags and drops pointer capture. The save round-trip causes a re-render mid-drag, which can drop pointer capture, which fires a fresh pointerdown on whatever's under the cursor — breaking one user gesture into multiple resize sessions.

3. The prop chain has 8 hand-offs for startGroupEdgeResize. Any one can drop the prop silently because TypeScript widens object literal types to satisfy the interface even when fields are missing. We just hit one (VideoEditorProvider.tsx was missing the field, surfacing as "snaps back to full shot length").

4. resizeGroupEdge has wrong semantics — it scales every child of the group proportionally when you drag an outer edge. Users expect: drag the right edge of the last clip → only that clip's duration changes. The current code shrinks every clip in the group.

5. Three different views of the same data (PinnedShotGroup → ShotGroup → PositionedShotGroup), each with its own filter that can drop the group from view, each with its own slightly different fields.

This entire conversation has been bug → fix → bug → fix for hours. Each fix passes type-check and tests. That's not a quality problem; it's a sign the model is so loose that structural bugs pass all the gates we have. The right answer is to stop patching and rebuild the model.

## The new model

Cohesion is a property of edit operations, not data shape.

### Data
pinnedShotGroups[i] = {
  shotId: string;
  trackId: string;
  clipIds: string[];     // ordered left-to-right by current spatial position
  mode?: 'images' | 'video';
  videoAssetKey?: string;
  imageClipSnapshot?: ...;
}

That's it. No start, no children, no offset, no duration on the group. The clips themselves carry all position/duration data via clips[k].at / clips[k].hold (or from/to/speed). One source of truth: the clips.

### Operations

Every user operation that touches a clip in a group is implemented as a direct clip mutation that preserves group cohesion intentionally:

- Drag clip body in a group → All clips in the group → Translate every clip in the group by the same delta. Already implemented in useClipDrag via deferred expansion — keep that pattern.
- Resize first clip's LEFT edge → First clip only → Change the first clip's at (and hold/source-time accordingly). Group's left edge follows. Other clips unchanged.
- Resize last clip's RIGHT edge → Last clip only → Change the last clip's hold/source-time. Group's right edge follows. Other clips unchanged.
- Resize an interior boundary (between clip[i] and clip[i+1]) → Two adjacent clips → Trade duration between them: clip[i]'s hold changes, clip[i+1]'s at AND hold change so clip[i+1]'s right edge stays put.
- Cross-track drag of a grouped clip → All clips in the group → All clips move to the new track preserving relative offsets. Update pinnedShotGroups[i].trackId.
- Pin a new shot group → (creates the group) → Sort the selected clip ids by at ascending and store as clipIds.
- Unpin / Deconstruct → (removes the group) → Just removes the entry from pinnedShotGroups. Clips are unchanged.
- Delete shot → All clips in the group + the group entry → Delete the clips, remove the pin.

No proportional scaling. Every operation only touches the clips it semantically should.

### Rendering

The shot border is a pure function of the clips' current positions:

function computeShotGroupRect(group, clips) {
  const groupClips = group.clipIds.map((id) => clipById.get(id)).filter(Boolean);
  if (groupClips.length === 0) return null;
  groupClips.sort((a, b) => a.at - b.at);
  const start = groupClips[0].at;
  const end = groupClips.at(-1).at + clipDuration(groupClips.at(-1));
  return { start, end };
}

No projection step, no validation pass that drops the group from view. If the clips are non-contiguous for some reason, render a degraded state but never silently lose the group.

### Save gating

Block auto-save while a drag or resize session is active. interactionStateRef.current.resize || .drag already exists; gate the save scheduler on it. When the interaction ends, flush any pending save. This eliminates the "drag resets mid-gesture due to save round-trip" symptom.

### Migration

Existing user data has the projection-shape pinnedShotGroups[i] = { ..., start, children: [...] }. The refactor must continue to load these correctly:

- On load (in repairConfig or equivalent), if a group has start + children, derive the legacy clipIds from children.map((c) => c.clipId) and DROP the start/children fields. The clip positions are already correct in clips[] from the previous projection.
- On save, only emit the new shape ({ shotId, trackId, clipIds, mode, ... }).
- Add a migration test that loads a projection-shape config and verifies it converts cleanly.

A smaller patch would keep two positional sources of truth alive, so the simplest structural fix is the rollback described in the brief: make `clips[]` the only source of timeline position/duration, reduce `pinnedShotGroups[]` to soft-tag metadata, and gate persistence while interactions are active.

## Phase 1: Canonical Model And Migration

### Step 1: Replace the persisted/runtime group shape with soft tags
**Scope:** Medium
1. Update `src/tools/video-editor/types/index.ts` so `PinnedShotGroup` only contains `shotId`, `trackId`, `clipIds`, `mode`, `videoAssetKey`, and `imageClipSnapshot`.

5. Prove the migration first with focused coverage in `src/tools/video-editor/lib/serialize.test.ts`, `src/tools/video-editor/lib/timeline-save-utils.test.ts`, and migration-adjacent tests before changing UI behavior.

### Step 2: Delete projection-era helpers and keep only lookup utilities
**Scope:** Medium

## Phase 2: Edit Operations Become The Cohesion Layer

4. Add focused tests for each operation and a sequence test that proves later edits are not silently reverted.

### Step 4: Keep grouped drag as clip translation, not group projection
**Scope:** Large
1. Update `src/tools/video-editor/hooks/useClipDrag.ts` and `src/tools/video-editor/lib/multi-drag-utils.ts` so grouped body drags and cross-track drags translate all member clips directly, reusing the current deferred expansion pattern as the canonical implementation.

### Step 5: Simplify group mutation entry points
**Scope:** Medium

This refactor eliminates the entire bug class structurally.\n\n## Why (the structural problem)\n\nThe previous \"first-class shot groups\" megaplan introduced a pinnedShotGroups[i] = { shotId, trackId, start, children: [{clipId, offset, duration}], clipIds, mode, ... } data shape. Any edit that bypasses the projection (a free-clip resize fallback, a stale closure, a missing prop) gets silently reverted because the next projection re-derives from children.\n\n2.
