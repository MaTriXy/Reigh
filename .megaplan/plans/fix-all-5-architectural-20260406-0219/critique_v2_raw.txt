        # Implementation Plan: Fix 5 Architectural Gaps in Video-Editor Effect System

## Overview

The video-editor effect system has five architectural gaps: (1) three effect sources merge silently with no dedup/priority, (2) the registry is non-reactive so components don't re-render when async registration completes, (3) no runtime error boundary for custom effects, (4) EffectCreatorPanel preview diverges from timeline wrapping pipeline, and (5) effect params are unvalidated `Record<string, unknown>` at apply-time.

**Architecture understanding:** `useEffectRegistry` (called in `VideoEditorProvider.tsx:21`) creates a provider-local `DynamicEffectRegistry` via `useMemo()` and mirrors it into a module-level singleton via `replaceEffectRegistry()` inside `useEffect()`. All render-time lookups (`lookupEffect`, `wrapWithClipEffects`, `EffectLayerSequence`) read from `getEffectRegistry()` — the module singleton. Remotion compositions re-render every frame via `useCurrentFrame()`, so once the module singleton is updated, clips pick up changes on the next frame. The core timing issue is that `replaceEffectRegistry()` runs in `useEffect()` (post-render), so the first render may hit a stale default registry.

**Scope:** `CustomEffectEditor.tsx` has no callers in the codebase (dead code) — excluded from this plan. The `customEffects` field in serialization/migration (`serialize.ts`, `migrate.ts`, `timeline-data.ts`) is a broader persistence concern outside the scope of these 5 architectural fixes.

## Phase 1: Foundation — Reactive Registry + Error Boundary

### Step 1: Make DynamicEffectRegistry reactive and fix first-render timing (`effects/DynamicEffectRegistry.ts`, `hooks/useEffectRegistry.ts`)
**Scope:** Small

1. **Add** to `DynamicEffectRegistry` class (`effects/DynamicEffectRegistry.ts`):
   - Private `_version: number = 0` and `_listeners: Set<() => void> = new Set()`
   - Arrow-function methods (for stable references): `subscribe = (listener: () => void): (() => void) => { this._listeners.add(listener); return () => this._listeners.delete(listener); }` and `getSnapshot = (): number => this._version`
   - Private `_notify()` method that increments `_version` and calls all listeners
   - Call `_notify()` at end of `register()`, `registerAsync()`, and `unregister()` — only when the state actually changed (skip if code-dedup early-return fires)

2. **Fix first-render timing** in `useEffectRegistry.ts:28-31`: Move `replaceEffectRegistry(registry)` out of `useEffect` and call it synchronously during render, immediately after `useMemo` creates the registry. This ensures the module singleton points to the provider registry before any child component renders. The async registration still happens in `useEffect`, but the registry object itself is available synchronously.

   ```typescript
   const registry = useMemo(() => new DynamicEffectRegistry(BUILT_INS), []);
   // Sync: make the module singleton point to this instance immediately
   replaceEffectRegistry(registry);
   ```

3. **Add** `useSyncExternalStore` subscription in `useEffectRegistry.ts` after the useMemo:
   ```typescript
   const _version = useSyncExternalStore(registry.subscribe, registry.getSnapshot);
   ```
   This forces `InnerProvider` (the only caller, `VideoEditorProvider.tsx:21`) to re-render when async registration completes. Since `InnerProvider` wraps all children, this propagation ensures the React tree sees updated state. Remotion compositions additionally re-render every frame via `useCurrentFrame()`, providing a second path to pick up changes.

### Step 2: Add EffectErrorBoundary with explicit fallback prop (`effects/EffectErrorBoundary.tsx`)
**Scope:** Small

1. **Create** `EffectErrorBoundary` class component with props `{ effectName: string; fallback: ReactNode; children: ReactNode }`. The `fallback` prop receives the content **before** effect wrapping — this is the key insight from the critique.
2. **On error:** `getDerivedStateFromError` sets `hasError: true`. `componentDidCatch` logs `[EffectErrorBoundary] "${effectName}" threw: ${error.message}`. `render()` returns `this.props.fallback` when errored, `this.props.children` otherwise.
3. **Reset** when `effectName` changes via `getDerivedStateFromError` returning `{ hasError: false }` when the key differs — or simply use a `key={effectName}` prop at the call site.

**Why fallback prop is needed:** In `wrapWithClipEffects`, `<Continuous>{wrapped}</Continuous>` is the effect wrapping clip content. If `Continuous` throws, an error boundary around it only sees the combined subtree. The boundary cannot extract the pre-wrapped `wrapped` content from `children`. By passing `wrapped` as `fallback` explicitly, the boundary can render the clip without the effect on error.

        - [CRITICAL] replaceEffectRegistry() during render is a side effect that violates React rules
  Step 1.2 proposes moving `replaceEffectRegistry(registry)` out of useEffect and calling it synchronously during render. This mutates module-level state (`let effectRegistry`) during render, which is a side effect. React requires render functions to be pure — side effects must go in useEffect, event handlers, or useMemo initializers. While the codebase doesn't currently use StrictMode, this would break under StrictMode (double-invocation) and violates React's concurrent rendering contract. The current useEffect placement is the correct React pattern. Instead, the plan should use `useMemo` to create AND replace the registry atomically: `const registry = useMemo(() => { const r = new DynamicEffectRegistry(BUILT_INS); replaceEffectRegistry(r); return r; }, [])` — this is idempotent and safe under double-render since it only runs once due to the empty deps.
