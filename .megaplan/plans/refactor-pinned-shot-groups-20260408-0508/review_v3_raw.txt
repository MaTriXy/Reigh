Refactor pinned shot groups in the video editor to be first-class timeline entities, eliminating an entire class of "shot group disconnects" bugs that come from fragile parallel cohesion logic spread across drag, resize, sync, and render code paths.

## Why

The video editor's pinned shot groups (the colored bordered boxes that visually group N timeline clips under a single shot label) are currently a "soft" abstraction layered on top of flat `row.actions` arrays. The shot group is just a `{ shotId, trackId, clipIds: string[] }` record stored in `data.config.pinnedShotGroups`. Cohesion (the property that all the group's clips stay contiguous on the timeline) is enforced by SEPARATE bespoke logic in every code path that mutates clip positions:

- Drag: `useClipDrag` does "deferred expansion" — when the user grabs one clip in a group, it pulls the group's other clips into the drag set. Then `multi-drag-utils.applyMultiDragMoves` calls `resolveGroupOverlaps` which treats moved clips as one shifted group.
- Resize: `TimelineCanvas` resize handlers compute a `getShotGroupAdjacent` neighbor and write a paired `ResizeOverride` so the immediate neighbor follows the dragged edge.
- Sync: `usePinnedGroupSync` runs on a debounce after the underlying Shot's `images[]` changes; it rebuilds the group's clips one-by-one and shifts subsequent clips on the row.
- Track move / cross-track drag: updates `pinnedShotGroups[].trackId` via `updatePinnedShotGroupTrackIds`.
- Delete / unpin / "deconstruct" / "delete shot": each touches `pinnedShotGroups` differently.

There's no shared "the group is the unit of editing" primitive. Each path reinvents cohesion. Bugs keep emerging in different paths with the same user-visible symptom: "the shot disconnects."

On top of that, the visual contiguity check in `TimelineCanvas.tsx` (`positionedShotGroups`, around lines 446-499) iterates `group.clipIds` IN ARRAY ORDER and checks `current.start < previous.start || current.start - previous.end > snapThreshold`. **This assumes `group.clipIds` is in left-to-right spatial order, but nothing actively maintains that invariant.** `pinGroup` stores `[...selectedClipIds]` (Set iteration order, i.e. click order), and `usePinnedGroupSync` rebuilds in `desiredImages` order from the database. Once array order disagrees with spatial order by even one position, the group's border vanishes — user reports "disconnected."

We just shipped two patches for this in the drag and resize paths. The patches work for their cases but they don't address the root cause: the shot group is not a real entity in the editing model.

## Goal

Make a pinned shot group a FIRST-CLASS timeline entity. Conceptually:

- The shot group has its own bounds (`start`, `end`) on the row.
- It contains child clips with their own positions inside that bound (or relative offsets).
- Drag, resize, selection, and serialization all see the group as ONE thing by default.
- The contiguity invariant is enforced BY CONSTRUCTION, not by validation. Children of a group can never become non-contiguous because they live inside a single owning entity that lays them out.
- Operations on individual clips inside a group are explicit, intentional, and rare (the per-clip context menu is already locked down to asset-state actions only when a clip is in a pinned group, per a recent change).

## Hard requirements

1. **Eliminate the "shot disconnects" bug class.** After this refactor, no drag, resize, sync, or other normal interaction can make a pinned shot group's children become non-contiguous. The only way to break a group is the explicit "Deconstruct shot" or "Delete shot" actions on the shot label menu.

2. **No regression in user-facing behavior** for the supported flows:
   - Pinning a group from a selection.
   - Dragging a group on its row (snaps to bounds, snaps near siblings, can hit timeline start at t=0).
   - Dragging a group cross-track (entire group moves to the new row).
   - Resizing a group's outer left/right edge (stretches/shrinks the group, content reflows).
   - Resizing an interior boundary inside a group (between child clips) — re-evaluate whether this is even meaningful in the new model. If kept, must preserve cohesion.
   - The shot label menu actions (Deconstruct, Delete, Switch to Final Video, Switch to Images, Update to Latest Video, Generate Video, Jump to Shot).
   - Right-clicking a clip inside a pinned group still shows only asset-state actions per the recent change.
   - The `usePinnedGroupSync` debounce still keeps the group's children in sync with the underlying Shot's `images[]`.
   - Final video and image modes still work.
   - Auto-shift on resize from the recent megaplan still works (or is replaced by a better mechanism in the new model).

3. **Persistence stays compatible.** The on-disk timeline config schema currently stores `pinnedShotGroups: { shotId, trackId, clipIds, mode, imageClipSnapshot? }[]`. Decide whether the new model can keep this on-disk shape (with internal restructuring), or whether a schema migration is needed. If a migration is needed, it must be backward-compatible (old configs load correctly).

4. **Remotion preview rendering stays correct.** The Remotion compositions render flat clips. The refactor must not break what Remotion sees — i.e., individual clips inside a group must still be addressable for rendering, even if the editing model treats them as children of a group.

5. **Tests pass.** All existing tests in `src/tools/video-editor/` must continue to pass, OR be updated with clear justification when behavior intentionally changes. New tests should cover the cohesion-by-construction property.

## Design questions the plan must answer

The planning phase needs to investigate the codebase and propose answers to all of these. Do not guess — read the code:

1. **Data model.** Which of these is the right abstraction?
   a) "Group action" embedded in `row.actions`: the row's actions array becomes a heterogeneous list of `TimelineAction | GroupAction`, where `GroupAction` has its own start/end and a `children[]` of TimelineActions. Drag/resize see the GroupAction as a single thing.
   b) "Group resolver" overlay: keep `row.actions` as flat TimelineActions, but introduce a `GroupResolver` that, given a clipId or set of clipIds, returns the group bounds and member list. Drag/resize hooks consult the resolver and operate on the whole group when any member is touched.
   c) Something else.
   Trade off: (a) is conceptually cleaner but ripples through many types and Remotion assumptions; (b) is less invasive but still leaves cohesion as a soft property enforced by callers.

2. **Selection model.** When the user clicks one clip inside a group, do they select (i) the clip, (ii) the group, or (iii) is there a hover/click distinction (e.g., one click = group, double click = clip)? The current behavior is "click selects the clip"; the recent context-menu change limits what that clip can do. Pick a model that makes "you cannot accidentally break a group by editing one of its clips" obvious.

3. **Internal layout.** Inside a group, children are positioned. Are positions stored absolutely (each child has its own row-relative start/end) or relatively (each child has an offset from the group's start)? Relative positioning makes "shift the whole group by N seconds" trivial and impossible to break; absolute positioning means children can drift relative to each other (the original bug). Strongly prefer relative.

4. **Interior boundaries.** Today, a user can resize the boundary between two child clips of a group via the auto-shift mechanism we added. Decide whether interior boundary resize is still allowed in the new model. If yes, define it precisely. If no, document why not (likely it should not be — the user already said they don't really want per-clip operations inside a group).

5. **Sync interplay.** `usePinnedGroupSync` watches the Shot's `images[]` and rewrites the group's clips. In the new model, this becomes "rewrite the children of the group in place." Specify how this interacts with in-progress drags/resizes (it currently debounces; does that still work?).

6. **Migration / persistence.** Define the on-disk shape and the load-time conversion if needed. Old configs must load.

7. **Snap & timeline-start clamp.** Today the bug-fix from a few minutes ago lets `findBestGroupStart` clamp the group to t≥0 as a unit. In the new model, the group's `start` would be clamped directly. Make sure that still happens in every drag/resize path.

8. **Reuse of existing infrastructure.** `multi-drag-utils.ts`, `resolve-overlaps.ts`, `useClipDrag.ts`, `useClipResize.ts`, `usePinnedShotGroups.ts`, and `useShotGroups.ts` all encode pieces of cohesion. Decide which become obsolete and which get repurposed. Avoid leaving dead code.

# Implementation Plan: Pinned Shot Groups as First-Class Timeline Entities (rev 3)

## Overview

Pinned shot groups are a sidecar `{ shotId, trackId, clipIds[], mode, imageClipSnapshot? }` record on `TimelineConfig`. Cohesion is enforced ad-hoc by drag deferred-expansion, resize paired-overrides, sync rebuild, and a contiguity check that wrongly assumes `clipIds` is in spatial order. The disconnect bug class follows.

**Approach (settled):** Hybrid — group-centric authoring with `start: number` + `children: { clipId, offset, duration }[]` (relative offsets), flat `TimelineConfig.clips[]` retained as the Remotion source of truth, projected from `group.start + child.offset` via a single `projectPinnedShotGroups(config)` normalizer. Cohesion-by-construction.

**Rev 3 fixes the three concrete root causes left from rev 2:**

1. **Keyboard delete/split on a group member is a no-op** (FLAG-002, issue_hints-1, callers-2). Rev 2 wrongly proposed rerouting to delete-shot; the brief is explicit that ONLY the shot-label-menu Delete/Deconstruct may break a group. ArrowUp/ArrowDown and `moveSelectedClipsToTrack` still reroute to whole-group moves because moving the group does not break cohesion.

2. **`useExternalDrop` builds the group inside its own loop** (FLAG-003, issue_hints-2, correctness-2, callers-1, all_locations-2, scope-3, scope, issue_hints-3) — using the `timeOffset` accumulator and `nextEdit.duration` it already tracks at `useExternalDrop.ts:148-169`. The new helper takes a `children` list directly; no need to read `workingData.config.clips`. The selection-pinning caller uses a thin convenience wrapper that derives children from `config.clips`.

3. **`isInteractionActive()` is a shared ref lifted into the editor state layer**, not on `useDragCoordinator` alone (correctness-1, all_locations-1). Both drag (in `useClipDrag`) and resize (in `TimelineCanvas`'s `resizeSessionRef`/`pendingOpsRef` lifecycle) write to it; `usePinnedGroupSync` reads it.

The hybrid model, projection wiring, clone helpers, Delete-shot semantics preservation, and broader edit surface from rev 2 are all preserved.

Scope: video-editor tool only.
