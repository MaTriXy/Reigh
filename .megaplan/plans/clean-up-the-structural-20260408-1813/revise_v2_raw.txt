Clean up the structural smells in the video-editor resize/preview architecture that have caused recurring bugs all session. The most recent symptom: when a user drags an interior boundary inside a pinned shot group, the live preview position is subtly wrong during the drag (boundary lags or jitters), but the committed position on release is correct. The root cause is structural — the live preview math and the commit math live in two different code paths that recompute the same answer in different files, the dual state (resizePreviewRef vs resizeOverrides) lets the rendered position drift one frame behind the synchronous source of truth, and the cursor-to-handle offset isnt captured anywhere so the boundary always trails the cursor by some pixels. None of these are bugs — theyre places where the architecture relies on parallel implementations agreeing, with no shared function or test enforcing it.

This is a focused structural cleanup, not a re-architecture. Five items, in priority order by leverage. Each item is testable in isolation.

State of completion BEFORE this megaplan: Coming out of the previous finish projection rollback megaplan, the data shape is fully soft-tag, the resize session model is unified (clip | group-child-boundary | group-edge), useClipDrag/useClipResize/usePinnedGroupSync/useSwitchToFinalVideo/useExternalDrop/useTimelineTrackManagement are all rewritten for the soft-tag model, the deprecated shims are gone, and 210/210 video-editor tests pass. The user has confirmed dragging an interior boundary ends up perfect on release — so the commit path is correct. The bug is purely in the live preview being subtly off from the committed result.

The 5 cleanups:

1. Extract computeResizeBoundary as a shared pure function (highest leverage). Currently TimelineCanvas.updateResize (the live preview) and useClipResize.onActionResizeEnd (the commit) both compute the new boundary position given a cursor delta — but in different files, with different intermediate variable names, and no shared function. Create src/tools/video-editor/lib/resize-math.ts with computeGroupChildBoundaryResize and computeGroupEdgeResize pure functions. Both live preview and commit call these. Add unit tests for clamping, direction, min-duration.

2. Capture the cursor-to-handle offset at session start. The resize handle is 8px wide. deltaSeconds = (clientX - initialClientX) / pixelsPerSecond assumes the cursor STARTED ON the boundary. Fix: at pointer-down, compute boundaryPixel = startLeft + (dir === left ? action.start : action.end) * pixelsPerSecond and cursorOffsetPx = clientX - boundaryPixel. Store on session. In updateResize: cursorBoundaryPixel = clientX - cursorOffsetPx; cursorBoundaryTime = (cursorBoundaryPixel - startLeft) / pixelsPerSecond. Apply for all three resize kinds.

3. Eliminate the dual resizePreviewRef / resizeOverrides state. Use useSyncExternalStore to subscribe to the ref, or a version counter. Eliminate the React state lag. SortableRow renders from the same source the commit reads.

4. Centralize pixel ↔ time conversion. Extract useTimelineScale({ scale, scaleWidth, startLeft }) returning pixelsPerSecond, pixelsPerSecondRef, timeToPixel, pixelToTime. All consumers use it.

5. Add an end-to-end resize integration test using @testing-library/react + fireEvent. Render real TimelineCanvas with shot group of 3 clips.
