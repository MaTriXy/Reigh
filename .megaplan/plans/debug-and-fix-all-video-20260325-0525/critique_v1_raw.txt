        # Implementation Plan: Debug and Fix Video Editor Bugs

## Overview

The video editor at `src/tools/video-editor/` is a full-featured timeline editor built on `@remotion/player` and `@xzdarcy/react-timeline-editor`. It has 7 known bugs ranging from a critical re-rendering loop to z-index and drag-drop issues. The core data flow is: `useTimelineData` (React Query polls + debounced save) → `useTimelineState` (composes context values) → 3 context providers → consumer components (`PreviewPanel`, `TimelineEditor`, `PropertiesPanel`).

**Key constraint**: The `resolvedConfig` object passed to `RemotionPreview` is the primary suspect for re-renders — it's derived from `data?.resolvedConfig` which is recreated on every poll cycle and every save. The Remotion `Player` remounts when `inputProps` identity changes.

**Repo shape**: ~630 LOC in `useTimelineData.ts`, ~843 LOC in `useTimelineEditing.ts`, ~280 LOC in `useTimelineState.ts`. Components are already `memo`'d in some places (`PreviewPanel`, `OverlayEditor`, `VideoEditorShell`).

## Step 2: Fix Player re-rendering — stabilize `resolvedConfig` identity
**Scope:** Medium — core fix for bug #1
**Files:** `useTimelineData.ts:471`, `useTimelineData.ts:398-430`, `useTimelineData.ts:432-461`, `RemotionPreview.tsx:31`

1. **Memoize `resolvedConfig` by signature** (`useTimelineData.ts:471`):
   ```typescript
   const resolvedConfig = useMemo(() => {
     return data?.resolvedConfig ?? null;
   }, [data?.signature]);
   ```
   This ensures `resolvedConfig` only gets a new identity when the config content actually changes (signature = `JSON.stringify`), not when the parent `data` object is recreated with identical content.

2. **Guard poll-sync effect** (`useTimelineData.ts:398-430`): The `shouldAcceptPolledData` function already checks `polledSig !== lastSavedSig`, but the issue is that after a successful save, `lastSavedSignature` is set to the saved signature, then the *next* poll returns the same data but `polledData.signature` equals `lastSavedSig`, so it's correctly rejected. **Verify** this path doesn't leak through. If it does, add an early return when `polledData.signature === dataRef.current?.signature`.

3. **Guard asset registry sync** (`useTimelineData.ts:432-461`): The `assetRegistryQuery.data` object gets a new identity every 2s poll. The effect already checks `nextData.signature === current.signature`, but still calls `buildTimelineData` (async) on every poll. **Add a ref** to cache the last `assetRegistryQuery.data` identity and skip if `===`.

4. **Stabilize `inputProps` in RemotionPreview** (`RemotionPreview.tsx:31`): Already has `useMemo(() => ({ config }), [config])`. Once `config` identity is stable from step 2.1, this is sufficient. **Verify** no remount.

5. **Remove temporary instrumentation** from Step 1.

**Success criterion**: Player plays continuously without stopping/restarting during idle editing. No visible flicker during debounced saves.

## Step 7: Fix play button blocked by overlay z-index
**Scope:** Small
**Files:** `PreviewPanel.tsx:29-64`, `RemotionPreview.tsx:105`, `OverlayEditor.tsx:291`

1. **Diagnose**: The layout in `PreviewPanel.tsx:30-64` is:
   ```
   <div class="relative flex">           ← container
     <div class="relative flex">          ← inner wrapper (line 31-32)
       <RemotionPreview />                ← has play controls at bottom (line 105-131)
       <OverlayEditor />                  ← absolute positioned, pointer-events-none root (line 291)
     </div>
   </div>
   ```
   The `OverlayEditor` root div is `pointer-events-none absolute` covering the entire video area. Individual overlay items are `pointer-events-auto`. The play button controls in `RemotionPreview.tsx:105-131` are inside the player container div, positioned `absolute inset-x-0 bottom-0`. The `OverlayEditor` div sits on top of the player in DOM order.

2. **Fix**: The play controls div already has `pointer-events-none` on the gradient wrapper and `pointer-events-auto` on individual buttons. The issue is stacking context: `OverlayEditor`'s absolute div covers the controls even though it's `pointer-events-none`, because overlay items (`pointer-events-auto`) intercept clicks in the bottom area.

   **Solution**: Add `z-10` (or `z-[10]`) to the controls bar in `RemotionPreview.tsx:105`:
   ```
   <div className="pointer-events-none absolute inset-x-0 bottom-0 z-10 flex ...">
   ```
   And ensure the OverlayEditor root doesn't have a z-index that overrides this (it currently doesn't — `OverlayEditor.tsx:291` has no z-index).

   **Alternative**: Move the controls div outside/after the OverlayEditor in the DOM, inside PreviewPanel. This is cleaner but requires extracting the controls from RemotionPreview.

**Success criterion**: Play/pause and skip-back buttons are always clickable, even when overlay items are near the bottom of the preview.

## Step 8: General stability audit — reduce unnecessary re-renders
**Scope:** Medium
**Files:** `useTimelineState.ts`, `VideoEditorProvider.tsx`, `TimelineEditor.tsx`, `RemotionPreview.tsx`

1. **Audit context value stability** in `useTimelineState.ts`:
   - The `editor` context value (`useMemo` at line 130) has **30 dependency entries** in its deps array. Many are stable refs (`dataRef`, `crossTrackActive`, etc.) but `dataHook.data`, `dataHook.resolvedConfig`, `dataHook.selectedClipId`, `playback.currentTime` all change frequently.
   - **Key problem**: `playback.currentTime` is NOT in the `editor` useMemo deps (good), but it IS in the `playbackValue` deps at line 269. This means every frame update during playback triggers a new `playbackValue` context object, which re-renders all `TimelinePlaybackContext` consumers.
   - **Fix**: `currentTime` updates during playback fire at ~30fps. The `PreviewPanel` and `OverlayEditor` consume `currentTime`. `OverlayEditor` uses it to determine visible overlays. This is necessary but expensive.
   - **Mitigation**: Use a ref + subscription pattern for `currentTime` in the playback context, or throttle `currentTime` state updates to ~10fps for the overlay (use `requestAnimationFrame` batching in `useTimelineSync`).

2. **Verify `RemotionPreview` doesn't remount** unnecessarily after Step 2 fixes. Add a `useEffect(() => console.log('[RemotionPreview] mounted'), [])` temporarily to confirm.

3. **Check `TimelineEditor` re-renders**: It consumes `useTimelineEditorContext()` which includes `data` and `resolvedConfig`. Every data change re-renders it. The `@xzdarcy/react-timeline-editor` `Timeline` component may handle this well internally. **Verify** by checking if the timeline flickers during playback.

4. **Stabilize the `useTimelineEditing` hook**: Many callbacks depend on `selectedClipId` and `selectedTrackId` (e.g., `handleAssetDrop` at line 501 depends on `selectedTrackId`). When selection changes, all these callbacks get new identities, which invalidates the `editor` context memo. **Consider** using refs for selection state in callbacks that don't need to trigger re-renders.

5. **Remove temporary instrumentation**.

**Success criterion**: During video playback, only `PreviewPanel`/`OverlayEditor` re-render per frame. `TimelineEditor` and `PropertiesPanel` don't re-render unless data changes. No visible jank.
